# 00 Repo Policy

## 1. 当前阶段
`main` 分支当前被定义为：

**论文与系统设计文档主分支**。

这意味着，主分支优先保证的是：
- 研究问题清晰
- 系统边界明确
- 模块规格完整
- 数据契约稳定
- 评估协议可执行
- 实验设计可复现

而不是“尽快塞进一些能跑的代码”。

## 2. 为什么清理了第一版代码
第一版 `src/` 与 `tests/` 的作用只是帮我们确认一个方向：
- Policy Graph 是合理的主轴
- Clarification-before-action 是必要能力
- Harness Runtime 是系统中心
- Experience Loop 可以作为持续进化入口

但那一版代码仍然存在以下关键缺陷：
1. 没有真实场景级 node schema
2. 没有 action contract 和 risk tier
3. 没有 trace schema 和 replay 机制
4. 没有 weighted compliance scorer
5. 没有 experience extraction 与策略更新协议
6. 没有实验输入输出契约

因此，那一版代码只能算“思路草稿”，不能算论文实现基线。

## 3. 主分支允许进入的内容
当前阶段，允许进入 `main` 的只有两类内容：

### A. 高置信文档
包括：
- 选题定位
- 文献矩阵
- 架构与模块说明
- 数据规范
- 评估协议
- 实验计划
- 模块 handoff

### B. 满足门槛的实现
只有当某个模块满足下面 5 个条件时，才能进入 `main`：
1. 有明确对应的模块 brief
2. 有输入输出契约
3. 有最小评估用例
4. 有 trace 设计
5. 有失败路径说明

## 4. 主分支当前不接受的内容
- 没有规格文档支撑的 demo 代码
- 只靠 prompt 黑盒完成的脚本
- 无法 replay 的 agent 运行逻辑
- 没有评估协议的原型
- 为了“看起来在开发”而堆的占位实现

## 5. 推荐的后续开发流程
后续建议采用如下顺序：

### Step 1
先读文档，确认模块边界和评估口径。

### Step 2
在新分支重建单个模块，不直接改主分支。

### Step 3
先补：
- 输入输出契约
- trace 字段
- 最小评估样例

### Step 4
再写代码。

### Step 5
通过离线评估与 replay 后再考虑合入。

## 6. 分支建议
- `main`：稳定文档与通过门槛的实现
- `exp/*`：单模块试验分支
- `eval/*`：评估和 judge 分支
- `sim/*`：模拟器与故障注入分支

## 7. 对后续模型的要求
后续不管是哪一个更强模型接手，都必须遵守一条原则：

**先对齐文档，再写实现。**

如果实现与文档冲突，以文档中明确写下来的契约与评估协议为准，而不是以某个旧脚本的行为为准。
