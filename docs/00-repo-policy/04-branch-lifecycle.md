# 04 Branch Lifecycle

## 分支类型
### main
用途：稳定文档与通过门槛的实现。  
要求：所有内容都应可作为“当前默认基线”被引用和继续开发。

### exp/*
用途：单模块试验。  
典型场景：
- Policy Graph schema 尝试
- Clarification 策略尝试
- Action Gating 规则尝试

### eval/*
用途：评估器、judge、replay、scoring 相关开发。  
要求：不得在没有样例数据的情况下空写评估逻辑。

### sim/*
用途：模拟器、故障注入、扰动场景。  
要求：必须明确每个场景的 expected path 与扰动标签。

## 生命周期
### 1. 创建分支
新分支创建前，必须先明确：
- 它服务哪个模块
- 它要解决哪个具体问题
- 它完成后的合并门槛是什么

### 2. 试验阶段
试验分支允许快速迭代，但不允许：
- 修改论文核心结论
- 绕开既有数据契约
- 省略 trace 与最小评估

### 3. 收敛阶段
准备合并前，必须补齐：
- 文档链接
- 输入输出定义
- 最小 case
- 评估结果
- 已知限制

### 4. 合并或废弃
分支只有两种命运：
- 合并入主分支
- 明确废弃并记录原因

不建议长期保留“半死不活”的实验分支。

## 命名建议
- `exp/policy-graph-v1`
- `exp/clarification-voi`
- `eval/replay-runner-v1`
- `sim/delay-fault-cases`

命名必须能让人一眼看出：
- 模块
- 主题
- 版本或目标
