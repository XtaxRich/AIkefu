# 03 Governance Principles

## 目标
本文件定义仓库治理的底层原则。后续所有实现、评估、实验和文档更新，默认都要服从这里的规则。

## 原则一：论文主问题优先
任何新增内容都必须回答：
1. 它是否直接服务论文主问题？
2. 它是否能被现有评估协议验证？
3. 它是否让系统更清晰，而不是更花哨？

如果答案是否定的，就不应进入主分支。

## 原则二：文档先于实现
实现不是第一产物，规格才是第一产物。任何模块开发都应先明确：
- 研究动机
- 模块边界
- 输入输出
- 失败模式
- trace 要求
- 最小评估样例

## 原则三：评估先于优化
没有评估协议的优化，默认视为无效优化。后续不允许出现“先写代码，之后再想怎么验证”的开发方式。

## 原则四：trace 优先于 demo
本仓库研究的是执行系统，不是演示系统。任何模块只要产生关键决策，就必须留下可追踪、可回放、可审计的过程信息。

## 原则五：低复杂度优先
第一阶段默认拒绝复杂多 agent 方案。优先采用：
- 单执行 Harness
- 显式 Policy Graph
- Clarification before action
- Action Gating
- Replayable Trace
- Non-parametric improvement loop

## 原则六：非参数持续进化优先
持续进化不是禁止，而是优先走低风险路径：
- 经验回流
- 策略记忆更新
- 澄清偏好更新
- 评估反馈驱动的外层配置更新

当前不把在线底模微调作为主分支默认方向。

## 原则七：主分支只保留高置信资产
主分支代表“当前最可信的研究基线”。因此：
- 可以保留高置信文档
- 可以保留通过门槛的实现
- 不保留占位 demo
- 不保留无法评估的黑盒脚本
- 不保留只为展示效果存在的临时代码
