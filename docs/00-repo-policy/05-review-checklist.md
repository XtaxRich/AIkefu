# 05 Review Checklist

## 适用范围
本清单适用于：
- 新文档进入主分支
- 新模块实现进入主分支
- 实验结论被写入仓库

## A. 研究一致性检查
1. 该内容是否直接服务论文主问题？
2. 是否与当前题目范围冲突？
3. 是否把系统重新带回“泛 AI 客服系统设计”方向？

只要第 3 条为“是”，默认不通过。

## B. 规格完整性检查
1. 是否有对应模块或文档定位？
2. 是否明确输入输出？
3. 是否明确失败模式？
4. 是否说明和其他模块的依赖关系？

## C. 数据与评估检查
1. 是否遵守既有数据对象定义？
2. 是否能留下 trace？
3. 是否有最小评估样例？
4. 是否能被 replay？
5. 是否能映射到既定指标？

## D. 实验有效性检查
1. 是否说明 baseline？
2. 是否说明 case 构造方式？
3. 是否说明结论边界？
4. 是否区分“离线有效”和“线上有效”？

## E. 主分支合并前最小证明物
未来任何实现想进入主分支，至少要附：
- 一份模块说明
- 一组最小样例
- 一份 trace 示例
- 一份评估结果摘要
- 一段已知限制说明

## F. 常见拒绝原因
- 只有代码，没有说明
- 只有 prompt，没有结构
- 只有 demo，没有评估
- 只有结果，没有 trace
- 只有结论，没有边界
