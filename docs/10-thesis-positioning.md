# 10 Thesis Positioning

## 1. 工作题目
**面向本地生活履约异常场景的执行 Harness 与持续进化服务智能体研究**

这是当前最推荐的工作题目。它比“AI 客服系统设计”更聚焦，比“通用 agent 平台”更稳，也更适合硕士论文落地。

## 2. 题目拆解
### 执行 Harness
关注智能体如何在高风险业务场景中真正“按要求办事”，而不是只会生成自然回复。

### 持续进化
关注系统如何从自然执行反馈、过程评估反馈和仿真反馈中持续优化外层策略，而不是依赖频繁改底模。

### 本地生活履约异常
把主战场限定在：催上门、迟到、改期、爽约四类场景，避免论文摊得过大。

## 3. 为什么这个题目成立
它同时满足四个条件：
1. 有真实业务痛点：履约异常直接影响投诉、转人工、用户体验和闭环效率。
2. 有明确研究对象：执行 Harness，而不是模糊的“更好客服”。
3. 有前沿支撑：policy adherence、process evaluation、clarification、simulator、continual improvement。
4. 有可实验性：可以构造 baseline、ablation、replay 和仿真评估。

## 4. 核心研究问题
### RQ1：静态执行
如何构建一个执行 Harness，使服务智能体在策略约束、信息不完备和工具依赖条件下，实现更高的过程合规性、异常恢复能力和真实业务闭环率？

### RQ2：持续进化
如何利用执行反馈、过程评估反馈和仿真反馈，使该系统在不频繁修改底模参数的前提下持续改进？

## 5. 论文的主对象
本文研究对象不是一个“大模型本体”，而是围绕模型构建的外层系统，核心包括：
- Policy Graph
- Clarification Router
- Action Gating
- Harness Runtime
- Process Trace / Eval Factory
- Experience Loop
- Simulator

## 6. 论文主贡献建议
建议只保留三条主贡献：

### Contribution 1
提出一个面向高风险服务场景的执行 Harness，把流程约束、动作边界、升级条件和轨迹审计显式结构化。

### Contribution 2
提出一个 clarification-before-action 机制，在信息不足时优先发起最小必要澄清，降低误动作与误路由。

### Contribution 3
提出一个基于过程反馈与经验回流的持续进化框架，使系统能在不频繁修改底模参数的前提下逐步改善。

## 7. 论文边界
### 论文做什么
- 做执行系统
- 做过程评估
- 做高风险服务场景
- 做非参数持续进化

### 论文不做什么
- 不做泛化开放域聊天
- 不做全渠道全链路一次性大统一
- 不做多 agent 群体复杂协作
- 不做在线底模强化学习主线
- 不做企业级完整部署

## 8. 为什么不直接写成“AI 客服系统设计”
因为那样太像行业方案，不像研究问题。它的问题会是：
- 研究对象不清楚
- 创新点太散
- 实验难以聚焦
- 更像项目总结而不是论文

而“执行 Harness + 持续进化”这个结构，既保留产业价值，也更符合学术表述。

## 9. 与未来 3–5 年主线的关系
这个题目并不把你锁死在“客服”标签里。它更大的方向是：

**服务结果智能 / 服务执行系统 / 高风险业务智能体运行时**

也就是说，这篇论文既能服务你当前业务，又能延伸到后续更大的职业主线。
